This little project is intended as a very small contribution to the World Statistics Day, 20/10/2010.

What this is about

The OECD exposes a very interesting and rich RESTful API (application programming interface) that returns SDMX data. SDMX is an international XML-based standard for the exchange of statistical data. Unfortunately, I couldn't find any official information about this programming interface. On the plus side, this interface almost auto-documents itself (although I may be exaggerating a bit here) and can be used for reference information.

The goal of this mini-site is to be a first step towards a fully documented SDMX API. It doesn't make any claim on being exhaustive, reliable, offical, etc.

Note: This is a personal initiative and is not related to my day job (although I do work at the OECD).

Todo list

As can be seen from the lists below, this is only just the beginning of what could become a very powerful way to expose the OECD's data to a very technical and knowledgable public, web developers.

missing things

  • provide JSON: XML is great, unfortunately due to security constraints it must come from the same domain as the application which requires additional techniques (such as a local proxy as used for this website) to be put in place. JSON does not have this drawback This is something the OECD should provide to make work easier for outside developers.
  • provide absolute values: when working with the unemployment figures (UNEMP), I've noticed that its description says something like "in '000". However, this is difficult to parse and it would be easier for the external developer to simply have the absolute values and then convert or present them in a manner that's appropriate for their application or website. This is also something that only the SDMX backend maintainer can provide.
  • a more polished framework: helper methods to translate the codes to their French or English equivalent; more options for the different outputs (language, format, layout, ...); more outputs (not just tables, charts, sparklines), csv, wiki, ...; output status codes where appropriate; error handling
  • an interactive sandbox: it's coming, it's just not ready yet, see Playground
  • access control: as with all APIs, a way to limit access to the data and to follow up on usage patterns to improve the API, some kind of authentication, typically via tokens should be put in place.

stuff that works

  • a basic framework of four different outputs:
    • single values,
    • sparklines,
    • tables,
    • charts.
  • a valid and working embed approach (see embed this!)
  • a basic reference of the methods that have been found by trial and error
